# PubLedge A11y Audit — 2026-04-19

## Summary

**Final state**: zero WCAG 2.1 AA violations across 17 pages, tested in both system dark mode and `.light-mode` class variant.

The site uses the civic palette introduced earlier today (navy `#0f2e5c` primary, crimson `#a31f34` accent, gold `#b8860b`, warm off-white `#fbfaf7`). Initial audit surfaced four classes of violation; all have been fixed via a combination of CSS updates (`assets/styles.css`), build-time HTML post-processing (`scripts/build-extras.js`), and stylesheet authored for the hand-crafted reference pages.

## Methodology

- axe-core 4.10.2 loaded via CDN into the running preview.
- Each target page loaded into an iframe at 1280×900; axe ran against the rendered DOM.
- Rules: `wcag2a`, `wcag2aa`, `wcag21a`, `wcag21aa` (plus `best-practice` spot-checks).
- Two variants per page: (1) OS dark mode (default for this machine), (2) `.light-mode` class forced.
- Manual inspection via `getComputedStyle` for the two cases where axe reports diverged from visual reality.

## Violations found + fixed

### 1. No `<main>` landmark on KaC-generated pages (moderate)

**Rule**: `landmark-one-main`, `region`.

KaC's `build.js` emits `<div class="container" id="main-content">` instead of `<main>`. Every KaC-generated page (home, containers, primaries, matrix, timeline, compare, about, entity details, bridges) failed axe's landmark checks.

**Fix** ([scripts/build-extras.js](scripts/build-extras.js) — new `patchA11y()` post-processor): regex swap of `<div ... id="main-content" ...>` → `<main ... id="main-content" ...>` and the corresponding closing `</div>` → `</main>` when it precedes `<footer>` or `</body>`. Applied at build time to 15 HTML files; no upstream KaC fork required.

### 2. Links in text blocks not distinguishable without color (serious)

**Rule**: `link-in-text-block` (WCAG 1.4.1).

Default styles had `a { text-decoration: none; }` with only color + hover-border differentiation. axe correctly flagged that users who cannot perceive color cannot locate links within paragraphs.

**Fix** ([assets/styles.css](assets/styles.css)):

```css
main a, .site-footer a, .prose a, article a, footer a {
  text-decoration: underline;
  text-decoration-thickness: 1px;
  text-underline-offset: 2px;
}
/* Cards, nav, CTAs, badges keep clean look */
.site-nav-link, .bridge-cta, .card-title a, .skip-link,
.stat-card a, .obligation-card > a, .search-results a,
.compare-list a, .breadcrumb a { text-decoration: none !important; }
```

### 3. Scrollable `<pre>` blocks not keyboard-focusable (serious)

**Rule**: `scrollable-region-focusable` (WCAG 2.1.1).

PROTOCOL prose on `/reference/protocol/` contains code blocks that overflow horizontally; without `tabindex` a keyboard user cannot scroll them.

**Fix** ([scripts/build-extras.js](scripts/build-extras.js)): `<pre(?![^>]*tabindex)` → `<pre tabindex="0"` across all built HTML.

### 4. Insufficient color contrast (serious)

**Rule**: `color-contrast` (WCAG 1.4.3).

Four sub-cases:

#### 4a. `.card-title a` in dark mode

`var(--navy)` (`#1a4180`) on dark panel `#161d2d` = 1.18:1. Fix: `.card-title a` now uses `--navy-ink` in light mode and an explicit `#e9ecf2` in dark mode.

#### 4b. `.stat-number` in dark mode

`var(--navy)` on dark bg, same issue. Fix: use `--navy-ink` default, `#e9ecf2` in dark mode.

#### 4c. `code.tpl-var` (template variable markers) in dark mode

General `code { color: #cbd5e1 }` rule overrode `.template-vars code` because of specificity (`:root:not(.light-mode) code` = 0,1,2 vs `.template-vars code` = 0,1,1). Light-gold text on light-gold background (`#cbd5e1` on `#fff6d6`) = 1.3:1.

Fix: `!important` on `.template-vars code, code.tpl-var` color + bg + border. Also: `build-extras.js` now emits `<code class="tpl-var">` on variable rows in the template detail pages so the dedicated selector matches.

#### 4d. `.status-badge.draft` in light mode

KaC emits per-page inline CSS from `project.yml` statuses. For `draft`, light-mode inline was `background: #b8860b; color: #fff` = 3.25:1 (white on mid-gold). The inline rule tied for specificity with my stylesheet rule but came later in the cascade, so it won.

Fix: `!important` on `.status-badge.draft|.reviewed|.published|.superseded` in `assets/styles.css` using soft-tinted pill backgrounds with dark ink text (e.g. draft: `#fff4cc` bg + `#5a4100` text = ~9:1).

### 5. Hand-authored reference/*/index.html palette + link styling

The `reference/protocol`, `reference/prior-art`, `reference/vocabulary`, `reference/registry` pages were hand-crafted before the civic repaint and kept their own inline `<style>` blocks with the legacy cyan accent (`#4fc3f7`) and no underlined prose links.

**Fix** ([scripts/build-extras.js](scripts/build-extras.js) `patchA11y()`): at build time, for any file matching `reference/*/index.html`:

- Replace `--accent: #4fc3f7` with `#1e4c8a` in the light-mode root and `#7fb0e8` in the dark-mode media block (civic navy for light, lighter blue for dark-bg contrast).
- Replace `--accent-bg: #0f3460` with `#0f2e5c`.
- Any remaining raw `#4fc3f7` → `#7fb0e8`.
- Inject a small a11y CSS block before `</style>` enforcing underline on prose links (`main a, article a, p a, li a, dd a, td a, th a, blockquote a`) and suppressing it on `.crumbs`, `nav`, `.chip`, `.views a`.

## Remaining best-practice notes (non-blocking, not WCAG violations)

- `region`: the `.skip-link` element sits outside any landmark. axe tags this as best-practice (not WCAG AA). It's intentional — skip-links are conventionally outside landmarks because they link *to* them. Leaving as-is.
- Home page pattern shelf (`pattern.html`) not audited; inherited KaC page with same shell as `/about.html` which passed.
