# Agent-Readiness Assessment: publedge.org

## Site overview

- Type: Public registry / open protocol reference site
- Platform: Static HTML generated by knowledge-as-code (`scripts/build.js`) hosted on GitHub Pages
- Subject: Fact-specific written interpretations between two parties (JIAs, RMAs, no-action letters, PLRs, CFPB advisory opinions, CFTC interpretive letters, SEC no-action letters)
- Content volume at time of audit: 14 legal instruments, 26 obligations, 14 mapping entries, 7 authorities, 5 templates
- Activity: Active — site bootstrapped 2026-04-18, under weekly contribution cadence

- Grade: **C** (77/100), "Needs Clearer Paths", failure mode "Can read but cannot route confidently"
- Pillars: access 100, readability 100, navigability 56, actionHandoff 40

  1. **fail** — public next-step paths weak (nav missing About/Services/Contact-class links)
  2. **fail** — no clear human handoff (contact/booking/inquiry)

## What agents currently see

### Strengths

- Every page is fully server-rendered HTML (static site generator, no SPA blind spot)
- 200 OK on every surface agents look for: `/llms.txt`, `/agents.json`, `/robots.txt`, `/sitemap.xml`, feeds, API, record detail pages
- Each record exposes four parallel representations: HTML page, `record.json`, Schema.org `LegalDocument` JSON-LD inline, PDF + OCR text
- Canonical URL hierarchy `/{country}/{region}/{authority}/{type}/{instance}/` is self-describing and stable
- Stable `identifier` on each record (old-style slug) for citations that predate the URL architecture
- `MANIFEST.yaml` SHA-256 hash integrity per skill-provenance.dev
- Split sitemap index + 7 per-section sitemaps (records, authorities, statutes, reference, templates, bridges, meta)
- Three feed formats side-by-side: `feed.xml` (RSS 2.0), `atom.xml` (Atom 1.0), `feed.json` (JSON Feed 1.1)
- MCP server exposes 13 tools including filtered listing, URL-based fetch, upcoming milestones, recently-changed

### Gaps (before remediation in this audit)

- `og:image` URL had double slash (`https://publedge.org//imgs/og.png`)
- `twitter:site` rendered as `[object Object]` (YAML empty-string mis-parse)
- Homepage `og:title` was `"Home"` rather than including site identity
- Homepage lacked its own JSON-LD (only record pages carried it)
- `robots.txt` used generic `User-agent: *` allow with no explicit AI-crawler entries
- No prominent "Use PubLedge" or contribute-handoff cards on homepage
- Registry-as-dataset semantics not declared (no `DataCatalog` anywhere)

## Unique opportunity

**PubLedge is a Dataset, not a business site.** The default agent-readiness rubric (contact page, services, CTAs) partially mis-scores a registry. The structural insight is that `DataCatalog` + `Dataset` + `DataDownload` schema on the homepage and index pages gives agents exactly the vocabulary they need to cite, query, and recommend the registry as a data source — rather than trying to route users to a "book a consultation" handoff that doesn't exist.

The remediation in this audit landed that vocabulary:

- Homepage now carries a `@graph` with `WebSite`, `Organization`, and `DataCatalog`
- `DataCatalog` enumerates three child `Dataset` nodes (Legal Instruments, Obligations, Coverage Matrix) with `DataDownload` distributions pointing at the JSON API
- `/matrix.html` is itself a `Dataset` with `DataDownload` distributions
- `/instruments.html` + `/obligations.html` + `/authorities.html` carry `ItemList` JSON-LD naming each entry

This lets Claude, ChatGPT, Perplexity, and Gemini surface PubLedge not as "a website about legal interpretations" but as "a queryable registry of legal interpretations with a stable schema." The MCP server (`mcp-server.js`) gives assistants direct tool-use access to the same data without scraping.

## Residual opportunities (not blocking)

- **Contact page.** Currently the contribute path is GitHub issues. A `/contact/` route aggregating the issue link, maintainer email (if desired), and a disclosure that PubLedge isn't legal advice would further strengthen handoff.
- **Per-record "Cite this record" block.** Inline code-block on record pages showing the stable identifier + canonical URL in BibTeX, Hansard, and Bluebook form.
- **Per-obligation `DefinedTerm` JSON-LD** (currently top-level list only; obligation detail pages could carry their own DefinedTerm with `inDefinedTermSet`).
- **`applies-to` bridge-page JSON-LD.** These pages exist but have no structured data yet.
- **JSON Feed category tagging.** Currently feed items lack `tags` (the field is `undefined` when source doesn't carry tags). Could derive from jurisdiction + authority + type.
- **Per-page `<meta name="author">`.** Currently inferred from Organization schema only.
