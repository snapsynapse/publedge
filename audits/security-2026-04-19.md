## OWASP Top 10 code pattern analysis

### A01: Broken Access Control — no findings

MCP server has no authentication and no per-user authorization, which is correct for its role (local stdio transport, read-only, exposes only repo content that is already public on publedge.org). No IDOR, no dev endpoints gated by environment checks.

### A02: Cryptographic Failures — no findings

No encryption in scope. `MANIFEST.yaml` uses SHA-256 for integrity (`shasum -a 256`) — modern, collision-resistant, appropriate for tamper detection.

### A03: Injection — see LOW-001, LOW-002

No SQL, no NoSQL, no command execution. All file access uses `path.join(ROOT, ...)` against pre-enumerated directory contents.

**LOW-001** — Unsafe HTML interpolation in generated client JS

- File: [scripts/build.js:708](scripts/build.js:708), [scripts/build.js:833](scripts/build.js:833), [scripts/build.js:849](scripts/build.js:849)
- Context: the build emits client-side JavaScript for the compare and containers pages that assigns to `innerHTML`. The interpolated values are either integer counts (line 708) or a string built from static status text plus user-selected checkbox IDs (line 849).
- Attack model: requires an attacker to (a) modify the repo to add a malicious container ID, or (b) trick a user into pasting a malicious URL fragment that the page reads. Neither path is exercised today — checkbox IDs come from server-rendered data and are stable.
- Recommendation: replace `innerHTML = ...` with DOM APIs (`textContent`, `document.createElement`) in the inlined client JS, or escape the interpolated values. Low priority because this is inherited KaC code and the content path is static.

**LOW-002** — Unescaped attribute values in search.js

- File: [assets/search.js:40-42](assets/search.js:40)
- Context: `results.innerHTML = matches.slice(0, 20).map(function (m) { return '<a href="' + href + '"><span class="search-type">' + (m.type || '') + '</span>' + escape(m.name) + '</a>'; })`. `m.name` is escaped; `href` and `m.type` are not.
- Attack model: `href` and `type` come from the build-generated `docs/assets/data.json`, which is built from trusted repo content. An attacker would need repo write access to inject a hostile `href` or `type`. Same blast-radius as directly pushing hostile code.
- Recommendation: escape `href` (attribute context) and `type` (text context) via the existing `escape()` helper or build the DOM node programmatically.

### A04: Insecure Design — no findings

The design uses static generation with no server-side state and no client-supplied identifiers used for trust decisions. Content integrity is enforced by `MANIFEST.yaml` hash tracking.

### A05: Security Misconfiguration — see INFO-001

**INFO-001** — MCP server has no rate limiting or connection auth

- File: [mcp-server.js](mcp-server.js)
- Context: the MCP server listens on stdio and answers tools/list, tools/call. There is no rate limit or authentication.
- Attack model: by design, stdio MCP servers are local subprocesses of the calling client (e.g., Claude Code). The security boundary is "only processes that can spawn this binary can talk to it." The same user who runs the shell has full repo access anyway.
- Recommendation: no change needed. Document the stdio-only, read-only, local-scope posture in `mcp.json` or a future SECURITY.md so operators don't expose it over a network.

### A06: Vulnerable and Outdated Components — no findings

Zero runtime dependencies. One vendored upstream (`vendor/gist/gistCore.ttl`, pinned snapshot, hashed). CC-BY 4.0. No `npm install` dependency chain to audit.

Recommendation: when GitHub Dependabot is enabled for the repo, it will still be useful for `@actions/*` versions if a GitHub Actions workflow is added. Not needed yet.

### A07: Identification and Authentication Failures — not applicable

No authentication surface in scope.

### A08: Software and Data Integrity Failures — see INFO-002

**INFO-002** — No SRI on external CDN scripts

- Context: `working-session/audits/a11y-2026-04-19.md` references loading axe-core from `cdnjs.cloudflare.com` during the a11y audit workflow. This is auditor tooling, not production code — no end-user page loads from a CDN. Production `docs/` site has no `<script src=>` pointing at external CDNs.
- Attack model: none in production. Auditor pulls axe-core at audit time only.
- Recommendation: no production change. If in the future the site decides to add client-side libraries, require `integrity=` SRI hashes.

**Integrity posture** (positive control): `MANIFEST.yaml` tracks SHA-256 of 24 canonical files (protocol prose, templates, instruments, obligations, authorities, schemas, mapping, vendored gist). `./scripts/validate-hashes.sh` verifies them. This is the project's integrity guarantee against tampering.

### A09: Security Logging and Monitoring — see LOW-003

**LOW-003** — No structured logging in MCP server

- File: [mcp-server.js](mcp-server.js)
- Context: the server prints protocol responses on stdout; errors go to stderr. There is no access log or audit trail of which tools were called or by whom.
- Attack model: not a vulnerability per se — stdio servers run as subprocesses of a trusted client — but a local compromise of the user account that spawns the server would not leave an audit trail.
- Recommendation: if operational visibility becomes valuable later, add a per-session log file written under a gitignored path (e.g., `.publedge-mcp.log`) with tool name + timestamp + arg summary. Not needed for v0.1.

### A10: Server-Side Request Forgery — no findings

No code in scope makes outbound HTTP requests using user-supplied URLs. The build pipeline reads local files only; `scripts/check-links.js` fetches only URLs already present in committed content (authored links, not user-supplied at runtime). MCP server does not make outbound requests.

## XSS review — build-extras.js template rendering

Reviewed `scripts/build-extras.js` `renderTemplates()` and `mdToHtml()`:

- `mdToHtml()` escapes all raw input via `escapeHTML()` before applying markdown inline regex (bold, italic, links, code). Escape-before-regex ordering is correct; no double-encoding.
- All frontmatter fields embedded in HTML (`title`, `id`, `kind`, `jurisdiction`, `status`, `disclaimer`, `slug`, variable names, variable descriptions) pass through `escapeHTML()`.
- `slug` is used in generated URLs without URL-encoding. In practice slugs are kebab-case ASCII (enforced by convention and by the schema's `pattern: ^[a-z]{2}(-[a-z0-9]+)+-tpl-[a-z0-9-]+$` for template IDs). A non-ASCII or special-char slug would not break security but could produce malformed URLs. Non-blocking.

**Conclusion**: template rendering is XSS-safe for any frontmatter content that currently exists or is specified by the schemas.

## Positive security controls

1. **Zero runtime dependencies.** Eliminates third-party supply chain attacks entirely for the built site.
2. **Hash-pinned integrity.** [MANIFEST.yaml](MANIFEST.yaml) tracks SHA-256 of all canonical content; [scripts/validate-hashes.sh](scripts/validate-hashes.sh) verifies on demand. Tamper detection is part of the protocol.
3. **Gist upstream pinned.** `vendor/gist/gistCore.ttl` is a pinned snapshot with its own `VERSION.md` refresh procedure — not a live dependency.
4. **Private content excluded.** `.gitignore` covers `_private/`, `.env*`, `.perplexity-research/`, session handoff notes. Verified no leaks in tracked files or git history.
5. **Read-only, local-only MCP server.** No network binding, no write operations, no auth needed because the boundary is process-local.
6. **All build output escaped.** `escapeHTML()` is applied before any HTML-context interpolation of repo-sourced strings in PubLedge-authored rendering code (`build-extras.js`).
7. **No secret storage.** No credentials anywhere in repo; operations that would need them (lawyer-snapshot send, future promo distribution) happen out-of-band.
8. **No `eval` / `Function` / `exec` anywhere.** Verified by grep across `scripts/`, `assets/`, `mcp-server.js`.
9. **HTTPS enforced on GitHub Pages.** Cert provisioned 2026-04-19; `https_enforced: true` in Pages config; all production URLs use HTTPS.
10. **Skill-provenance manifest mechanism** adapted from skillprovenance.dev — integrity as a first-class protocol feature, not bolted on.
